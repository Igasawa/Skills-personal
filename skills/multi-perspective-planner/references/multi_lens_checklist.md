# 複眼検証チェックリスト（決定可否チェック）

## 1) 依存関係
- [ ] 事業前提（目的・KPI）が定義され、同意者が明確化されているか  
- [ ] 外部システム/API/インフラ依存が明示されているか  
- [ ] 内部リソース（担当者、予算、時間）との整合が取れているか  
- [ ] 依存先の障害・遅延時の代替手段があるか  

## 2) リスク
- [ ] 技術リスクが特定され、影響度・発生確率と対策があるか  
- [ ] オペレーションリスク（運用負荷、監視漏れ、権限不足）が洗い出されているか  
- [ ] コンプライアンス/セキュリティ観点の確認項目があるか  
- [ ] 停止条件とエスカレーション先が定義されているか  

## 3) 期限・SLA
- [ ] 主要マイルストーンに期限を持つか  
- [ ] 重要タスクの最短・最長見積（レンジ）があるか  
- [ ] 優先度が高い順に実施順が定まっているか  
- [ ] 進捗報告タイミングが定義されているか  

## 4) 監査証跡
- [ ] 決定理由と前提条件が明記されているか  
- [ ] 変更履歴・合意履歴の保存先が示されているか  
- [ ] 設計・実装・検証の主要決定に誰が責任者か示されているか  

## 5) テスト観点
- [ ] 事前条件と受け入れ条件が一致しているか  
- [ ] 品質ゲート（テスト種類/実施条件/期待結果）が定義されているか  
- [ ] テスト未完了時のフェールセーフがあるか  
- [ ] 本番移行条件が検証可能で明文化されているか  

## 6) 運用移行
- [ ] 運用引き継ぎの順序が定まっているか  
- [ ] ロールバック手順が実行可能な粒度で記載されているか  
- [ ] 運用監視イベント（例: 失敗時通知）が具体化されているか  

## 7) 停止条件 / エスカレーション
- [ ] 停止条件（例: 重大障害、予算超過、期限遅延）を列挙しているか  
- [ ] コンタクト先・判断基準・期限を示したエスカレーション条件があるか  
- [ ] 再開条件を明文化しているか
